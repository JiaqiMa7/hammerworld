## 四、隨機組合 + AI評估流水線

### 組合生成（防重複）

```python
def generate_combination(method_matrix, problem_matrix, user_seed):
    # 1. 基於區塊高度+用戶地址生成確定性隨機種子
    seed = hash(block_height + user_address + nonce)
    
    # 2. Fisher-Yates 打亂兩個矩陣
    shuffled_methods = fisher_yates_shuffle(method_matrix, seed)
    shuffled_problems = fisher_yates_shuffle(problem_matrix, seed + 1)
    
    # 3. 從打亂後的矩陣中按序配對
    #    而非隨機抽取，確保每輪掃描覆蓋不同區域
    pairs = []
    for i in range(batch_size):
        m = shuffled_methods[(nonce + i) % len(shuffled_methods)]
        p = shuffled_problems[(nonce + i * 3) % len(shuffled_problems)]
        pairs.append((m, p, i))
    
    return pairs
```

### AI多維度評估

每個組合由AI從以下維度評分（1-10）：

```
維度           評分標準                        高分含義
────────────────────────────────────────────────────
優雅度         方案簡潔、對稱、自洽             牛頓定律級別的簡潔
奇怪度         反直覺、出人意料、挑戰常識       量子力學級別的顛覆
人類可行性     現有技術/資源可實現               10年內可工程化
AI可行性       AI系統可自主執行                 不需要人類介入
新穎度         與已知方案的差異程度              從未被提出過
類比距離       方法來源領域與問題領域的跨度      跨界越遠越有趣
縮放潛力       小規模驗證後可放大的程度          可指數級擴展
副作用         方案引發新問題的嚴重性            負面影響越小越好
```

**關鍵設計：不對稱閾值**
- 任一維度 ≥ 8.0 即可上傳（而非要求平均分高）
- 這保證了「怪但有趣」的想法不會被埋沒

## 六、區塊鏈緩衝區（提交 → 分類 → 上鏈）

### 設計動機

AI分析結果不直接發佈到網頁，而是先進入鏈上緩衝區。原因：
- 防止未經審核的垃圾內容直接出現在排行榜
- 將「領域分類」工作開放給社區，降低中心化審核成本
- 為系統提供又一種「貢獻即挖礦」的途徑

## 七、系統維護獎勵（比特幣式算力貢獻）

### 7.1 安全漏洞與Bug賞金

```
漏洞等級        賞金倍數     範例
────────────────────────────────────
嚴重 (Critical)   100x      智能合約漏洞、代幣盜竊
高 (High)          50x      排行榜操縱、付費繞過
中 (Medium)        20x      P2P同步攻擊、數據篡改
低 (Low)            5x      UI漏洞、顯示錯誤
```

提交流程：
```
發現者提交漏洞報告 → 核心團隊驗證 → 確認等級 → 發放賞金
                                    │
                                    └── 漏洞修復後公開（榮譽榜記錄）
```

### 7.2 投毒檢測（AI Poisoning Detection）

```
目標: 檢測惡意用戶提交的虛假/誤導性AI分析

檢測任務:
├── 檢查AI分析是否存在幻覺（引用不存在的研究）
├── 檢查評分是否明顯異常（全部滿分或全部零分）
├── 檢查分析文本是否與組合無關（隨機灌水）
├── 檢查是否存在提示注入嘗試
└── 檢查模型簽名是否有效（防偽造AI分析）

檢測者工作:
├── 隨機抽樣緩衝區中的提交
├── 運行檢測規則（部分自動化 + 部分人工判斷）
├── 標記可疑提交 → 觸發質押罰沒
└── 獎勵: 每檢測到並確認的投毒提交 → D 代幣
```

### 7.3 矩陣合規檢測（Matrix Compliance Check）

```
目標: 確保用戶上傳的方法/問題符合質量標準

方法矩陣檢測:
├── 是否與已有方法重複（相似度檢查）
├── 描述是否清晰可執行（非空泛口號）
├── 分級是否合理（Level 1-4）
├── 跨領域標記是否準確
└── 是否包含惡意內容

問題矩陣檢測:
├── 問題是否為真實存在的科學/技術問題
├── 描述是否具體（非"解決世界和平"類空泛問題）
├── TRIZ標準化是否正確（由TRIZ Agent輔助驗證）
├── 領域分類是否準確
└── 是否與已存在的問題重複

獎勵: 每審核一條 → C 代幣
      審核結果與多數共識一致 → 額外獎勵
```

### 代幣消耗場景

```
消耗場景                    費用
──────────────────────────────────────────
查看AI詳細分析               N 代幣
  分配: 80%分析者, 10%組合發現者, 10%協議費
解鎖排行榜（付費模式）        每榜 P 代幣
解鎖隨機抽取查看             每抽 Q 代幣（低於定榜）
發起問題分析懸賞             自定賞金
購買高分方法矩陣             市場定價
```

### 抗垃圾機制

```
防範措施:
├── 提交需質押代幣（低質量提交被罰沒）
├── AI分析需附模型簽名+推理哈希
├── 投毒檢測節點持續掃描
├── 矩陣合規檢測節點審核新提交
├── 信譽系統：持續提交低分內容 → 質押要求提高
└── 查看者評分：付費查看後可對分析質量打分
      → 影響該分析者的信譽和未來收益
```

【按領域】
├── 醫學與健康 × 各方法排行榜
├── 能源 × 各方法排行榜

【交叉篩選】

```

### 兩種查看模式

```
模式A: 固定排名查看
┌────────────────────────────┐
│ 支付 P 代幣                 │
│ 解鎖「AI可行性榜 Top 50」    │
│                            │
│ 查看: 第1-50名 全部內容      │
│ 有效期: 永久                 │
└────────────────────────────┘

模式B: 隨機抽取查看 (推薦)
┌────────────────────────────┐
│ 支付 Q 代幣 (Q << P)        │
│ 從「奇怪度榜」隨機抽取 10 位  │
│                            │
│ 特點:                       │
│ ├── 可能抽到冷門寶藏          │
│ ├── 費用低, 鼓勵探索          │
│ └── 每次抽取結果不同          │
│                            │
│ 防重複:                     │
│ └── 用戶已抽過的idea不會重複出現│
│     直到該用戶遍歷過整個榜單    │
└────────────────────────────┘
```

**為何推薦隨機模式？**

- 固定排名會導致馬太效應（前10名佔據所有注意力）
- 隨機抽取讓長尾idea也有被發現的機會
- 符合TRIZ核心思想：突破性創新往往來自意想不到的組合
- 較低的費用降低了探索門檻

## 九、榮譽系統

### 9.1 實現榮譽（Implemented Ideas）

```
流程:
1. 用戶提交申報材料
   ├── 證明基於平台某idea實現了創新
   ├── 論文/專利/產品鏈接
   └── 描述idea如何指導了實現

2. 社區驗證
   ├── 多位驗證者審核申報材料
   ├── 確認idea與實現之間的因果關係
   └── 達到共識 → 進入榮譽流程

3. 系統操作

## 十三、矩陣商城（Matrix Marketplace）

### 設計動機

替換原有「提交→審核→合併」的單向流程。改為用戶創建**命名的 method/problem collection**，
其他用戶可瀏覽、加星、導入到自己的 mine 任務。

### 數據模型

```
method_collections (id, name, description, category, creator, stars, import_count, methods_json, created_at)
problem_collections (id, name, description, category, creator, stars, import_count, problems_json, created_at)
collection_stars (collection_type, collection_id, starrer, starred_at) -- 防雙重加星
```

### 推薦邏輯

- **方法推荐**：`ORDER BY stars DESC` — 星越多越靠前
- **問題推荐**：`ORDER BY import_count ASC` — 被導入越少越靠前（鼓勵探索冷門問題）

### 導入機制

```
python3 -m src.cli.main mine \
  --methods-collection "Creative Methods Pack" \
  --problems-collection "Energy & Climate Problems"
```

導入時自動 `import_count += 1`。

## 十四、數學研究專區（Math Research Zone）

### 設計動機

為數學未解問題建立專屬研究區域，以特定數學問題為最上層劃分（如黎曼猜想），
Peer 可申請創建問題區。訪問需通過「閘門機制」——運行數學工具庫與問題的隨機組合 + AI 分析來解鎖。
解法按「最大連續正確步驟數」排名，支持 Fork 協作。

### 層級結構

```
Math Problem (Top-level, e.g., Riemann Hypothesis)
  ├── Method Collection A (e.g., Complex Analysis Tools)
  │     ├── Solution 1 (Alice) - max_correct_step: 15
  │     ├── Solution 2 (Bob, forked from #1) - max_correct_step: 18
  │     └── Solution 3 (Charlie) - max_correct_step: 5
  ├── Method Collection B (e.g., Fourier Analysis)
  │     └── Solution 4 (Dave) - max_correct_step: 10
  └── Method Collection C (e.g., Algebraic Topology)
        └── Solution 5 (Eve) - max_correct_step: 3
```

### 閘門機制（Gate Mechanism）

```
用戶 → math-mine → 加載問題 + 數學方法庫 → 隨機組合 → AI 分析 →
成功結果存入方法池 → 自動授予訪問權（math_access_log）→ 解鎖該 (problem, method) 區域
```

成功挖礦結果不再寫入全局排行榜 `combinations` 表，而是存入問題專屬的 `math_method_pool` 表。每個問題有獨立方法池，記錄方法名稱、完整 AI 分析 JSON、最佳分數/維度、礦工地址。

Web unlock 頁面作為備用通道（手動輸入 combo_id 解鎖）。

### 數據模型

```
math_problems (id, title, description, category, creator, status, created_at)
math_solutions (id, problem_id, method_collection_id, user_address, parent_solution_id,
                steps_json, max_correct_step, seed_combo_id, seed_analysis_json,
                created_at, updated_at)
math_access_log (id, problem_id, method_collection_id, user_address, combo_id,
                 analysis_json, created_at)
math_method_pool (id, problem_id, method_collection_id, method_name, method_data,
                  analysis_json, best_score, best_dimension, miner_address, stars,
                  created_at)
math_method_pool_stars (method_pool_id, starrer, starred_at)
math_step_stars (solution_id, step_num, starrer, starred_at)
```

### 方法池（Method Pool）

成功挖礦結果不再寫入全局 `combinations` 表，而是存入 `math_method_pool`：

```
成功挖礦 → add_to_method_pool(problem_id, method_collection_id, method, analysis_json,
              best_score, best_dimension, miner_address)
         → grant_math_access(...) 自動授予訪問權
         → get_root_node(...) 自動創建根節點
```

方法池特點：
- **每個問題獨立**：`get_method_pool(problem_id)` 返回該問題的所有成功挖礦方法
- **含完整 AI 分析**：`get_method_pool_entry(pool_id)` 返回原始分析 JSON
- **Star 評價**：`toggle_method_pool_star(pool_id, starrer)` 切換標星狀態

### 自動樹擴展（Auto-Tree Expansion on Submit）

提交解法時，若該方法存在於方法池中，自動在 MCTS 樹根節點下創建第一層子節點：

```
math-submit → 檢查方法池 → 是否存在已挖礦方法？
  ├─ 是 → 檢查樹中是否已有同名 action_label？
  │    ├─ 無 → create_tree_node + create_tree_edge (parent=root)
  │    └─ 有 → 跳過（避免重複）
  └─ 否 → 正常提交，不創建樹節點
```

這使得方法池與 MCTS 樹之間建立了自動連接：挖礦發現方法 → 使用該方法提交解法 → 方法自動成為樹的第一層子節點。

### 解法步驟格式

```json
[
  {"step_num": 1, "content": "定義 Riemann zeta 函數...", "verified": true},
  {"step_num": 2, "content": "應用圍道積分...", "verified": true},
  {"step_num": 3, "content": "嘗試約束誤差項...", "verified": false}
]
```

`max_correct_step` = 最後連續 verified=true 的步驟數。

### 排名規則

同一 (problem, method) 區域內，解法按 `max_correct_step DESC` 排名。

### Fork 工作流

用戶可 Fork（pull）他人解法作為起點繼續工作，後續有成果時 submit 自己的更新。
`parent_solution_id` 記錄派生關係。

### CLI 命令

```bash
# 閘門解鎖（隨機組合 + AI 分析 + 自動授權）
python3 -m src.cli.main math-mine --problem-id 1 --methods-collection "Complex Analysis" --batch 3

# 提交/Fork 解法
python3 -m src.cli.main math-submit --problem-id 1 --method-collection-id 3 --steps-json '[...]' --parent-id 5

# 列出方法池
python3 -m src.cli.main math-pool-list 1

# 標星方法
python3 -m src.cli.main math-star-method 1 --address 0xALICE

# 標星步驟
python3 -m src.cli.main math-star-step 1 3 --address 0xALICE
```
